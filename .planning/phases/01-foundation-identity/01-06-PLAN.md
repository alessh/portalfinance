<objective>
HUMAN-UAT Test 4 surfaced two visual defects in the password-reset email:
(1) the CTA rendered as `[/api/auth/reset/confirm?token=...]Redefinir senha` —
react-email's plaintext fallback formats `<Button href>` with no scheme/host as
`[<href>]<label>`; (2) the path pointed at the API route rather than the UI page.
Commit 486a98a fixed the password-reset URL but the same commit message flagged an
**unfixed adjacent bug** at `src/app/api/auth/login/route.ts:196` (account-unlock
unlock_link). Auditing the unlock route reveals its shape `${env.NEXTAUTH_URL}/api/auth/unlock?token=`
is in fact correct — `GET /api/auth/unlock` validates the token and redirects to
`/unlock?result=ok|expired` (the UI page at `src/app/(auth)/unlock/page.tsx`). So the
account-unlock URL is NOT a code bug. But the lack of a regression test means a
future refactor could reintroduce the password-reset class of bug on any of the three
transactional templates without anyone noticing until prod UAT.

Purpose: Lock in the URL contract for every transactional email with **two layers of
defense** — unit tests on rendered email HTML, plus an integration test on the enqueued
payload shape — so the HUMAN-UAT Test 4 visual bug class cannot reach production again.
This is pure regression coverage; no production code changes.

Output: Three React-Email render unit tests (one per template) and one integration
test that drains enqueue payloads and asserts URL shape.
</objective>

- password-reset reset_link  → `${env.NEXTAUTH_URL}/reset/confirm?token=<base64url 32+ chars>`

URL forbidden patterns (must NEVER appear in any rendered email or any enqueued payload):
- Leading `/` followed by anything (relative URL)
- `http://` (must be https in production)
- `localhost` or `127.0.0.1`
- The literal string `${env.NEXTAUTH_URL}` (template-literal not interpolated)
- The literal string `undefined` (env var was missing)

<important_note>
**Why no env-module-runtime assertion (W2).** A previous draft of this plan included a
fourth integration test that asserted `(await import('@/lib/env')).env.NEXTAUTH_URL === 'https://portalfinance.app'`
via dynamic import inside the test body. That test cannot work reliably: `tests/setup.ts`
transitively imports `@/lib/env` before any per-test `beforeAll()` runs, so the env
module's parsed cache is locked at parse time. The dynamic re-import in the test reads
the cached value, not the test-set `process.env.NEXTAUTH_URL`. Tests 1-3 below already
prove the URL contract end-to-end through the actual HTTP route → enqueue payload
path — that IS the real contract, and it's what production reads. We dropped the
env-module assertion rather than work around it; the cost (an extra contrived test)
exceeded the value (no additional coverage of any real defect class).
</important_note>

   import { describe, it, expect } from 'vitest';
   import { render } from '@react-email/render';
   import { PasswordReset } from '@/emails/PasswordReset';
   import * as React from 'react';

   const URL_FIXTURE = 'https://portalfinance.app/reset/confirm?token=abc123def456ghi789jkl0';
   const EXPIRES = new Date(Date.now() + 60 * 60 * 1000);

   describe('PasswordReset email template', () => {
     it('renders a single anchor with absolute https href matching reset_link', async () => {
       const html = await render(React.createElement(PasswordReset, { reset_link: URL_FIXTURE, expires_at: EXPIRES }));
       const anchorRe = new RegExp(
         `<a\\b[^>]*\\bhref="${URL_FIXTURE.replace(/[.*+?^${}()|[\\]\\\\]/g, '\\\\$&')}"`,
         'i',
       );
       expect(html).toMatch(anchorRe);
       expect(html).toContain('Redefinir senha');
       expect(html).not.toContain('[https://');
       expect(html).not.toContain('[/');
     });

     it('plaintext fallback contains the absolute URL and no markdown-bracket leakage', async () => {
       const text = await render(
         React.createElement(PasswordReset, { reset_link: URL_FIXTURE, expires_at: EXPIRES }),
         { plainText: true },
       );
       expect(text).toContain(URL_FIXTURE);
       expect(text).not.toMatch(/\[https?:\/\/[^\]]+\]\S+/);
       expect(text).not.toContain('[/api/');
     });

     it('does not leak unsubstituted template-literal placeholders', async () => {
       const html = await render(React.createElement(PasswordReset, { reset_link: URL_FIXTURE, expires_at: EXPIRES }));
       expect(html).not.toContain('${env.');

     });
