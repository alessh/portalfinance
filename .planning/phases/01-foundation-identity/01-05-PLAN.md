<objective>
Phase 01 HUMAN-UAT Test 4 surfaced a production blocker: a password-reset request from
deployed prod (https://portalfinance.app) targeting `bounce@simulator.amazonses.com`
produced **zero log lines anywhere** — no `password_reset_email_sent` from the worker,
no `ses_bounce_received` on web. The worker IS registered for `email.password_reset`
(verified at `src/jobs/worker.ts:31`), so this is not a missing-handler bug. The real
defects in the mailer pipeline are:

1. **`src/lib/mailer.ts:124-131` builds `SendEmailCommand` WITHOUT `ConfigurationSetName`.**
   The Copilot manifest sets `SES_CONFIGURATION_SET=portal-finance` on both web and
   worker tasks (`copilot/web/manifest.yml:49`, `copilot/worker/manifest.yml`), but
   the mailer never reads it. Without the configuration set on the send command, SES
   does not publish send / bounce / complaint events to the SNS topic — which means
   the bounce-simulator address never appears in `ses_suppressions` and no
   `ses_bounce_received` log fires. The configuration set is wired in env / IAM /
   manifest but the code never attaches it.
2. **`sendEmail()` returns `{ messageId, suppressed }` silently on success.**
   The worker only logs on caught exceptions (`passwordResetEmailWorker.ts:32-36`).
   On a successful suppression, dev-mode skip, or actual SES accept, the only log
   emitted is the dev-mode warn (when creds are absent). There is no
   `email_send_attempt` log emitted by the mailer itself — so production ops cannot
   answer "did this send happen, and what was the outcome?" from logs alone.
3. **No plaintext alternate.** `SendEmailCommand` ships only `Body.Html`; some email
   clients downgrade to plaintext and render the HTML literally — this is a
   contributing factor to the secondary visual bug in HUMAN-UAT Test 4 (button
   appears as `[<href>]label`). Adding a `Body.Text` alternate (rendered via
   `@react-email/render` with `plainText: true`) gives clients a clean fallback.

Purpose: Make the password-reset (and unlock + DSR) email pipeline **observable in
production** — every send attempt produces a structured log with enough metadata
to diagnose without re-running the flow. Wire the SES configuration set so bounce
events actually reach the existing webhook. Add a regression integration test that
runs the entire `/api/auth/reset/request` → worker → mailer flow and asserts both
the log shape and the SES command shape so this class of bug cannot reach prod again.

Output: Updated mailer + workers with structured logging, ConfigurationSetName
attached, plaintext alternate, plus a unit test suite for the mailer (mocked DB), a
DB-backed integration test for the suppression guard, and an integration test
covering the full password-reset enqueue → handler → mailer pipeline.
</objective>
