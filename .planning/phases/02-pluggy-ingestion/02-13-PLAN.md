Close codex review concern **#4 (HIGH — Transfer detection can over-pair ambiguous transactions)**.

Current shipped behavior at `src/jobs/workers/transferDetectorWorker.ts` (Wave 4 plan 02-05): the `candidates` CTE produces ALL pairs satisfying the 4 invariants (same |amount|, opposite type, different account, ≤3 days apart). When one debit matches multiple credits of the same amount within 3 days, the UPDATE assigns transfer_pair_id deterministically by row evaluation order BUT the choice of which credit pairs with which debit is non-deterministic across runs (Postgres planner choice).

Symptoms:
- A transaction may appear in multiple pairs in the candidates CTE.
- The UPDATE may produce different transfer_pair_id values across runs of the same data (planner-dependent).
- The "is at most one pair" invariant is not enforced.

Fix:
1. Replace the candidates CTE with a deterministic ranking using ROW_NUMBER() OVER (PARTITION BY t1.id ORDER BY ABS time-delta ASC, t2.id ASC) — for each candidate-debit, pick exactly one credit (the closest in time; ties broken by id lex order).
2. Apply the same ranking from the t2.id side to ensure global 1-to-1 (mutual best match).
3. Document the algorithm inline + add 3 new tests proving the invariant.

Output: 1 file modified + 3 new tests.

**Step 1 — Replace the SQL inside `transferDetectorWorker.ts` execute() block:**

Replace the existing candidates CTE with a 3-CTE pipeline that enforces mutual best-match:

```sql
WITH same_user_pairs AS (
  -- All (debit, credit) candidate pairs satisfying D-33's 4 invariants.
  -- Note: t1.id < t2.id NOT applied here -- we want both directions because
  -- the asymmetry between t1 and t2 carries different semantics now (debit vs credit).
  SELECT
    debit.id  AS debit_id,
    credit.id AS credit_id,
    ABS(EXTRACT(EPOCH FROM (debit.posted_at - credit.posted_at))) AS delta_seconds
  FROM ${transactions} debit
  INNER JOIN ${transactions} credit
    ON debit.user_id = credit.user_id
    AND debit.user_id = ${user_id}
    AND debit.is_transfer = false
    AND credit.is_transfer = false
    AND debit.type = 'DEBIT'
    AND credit.type = 'CREDIT'
    AND debit.account_id <> credit.account_id
    AND debit.amount = credit.amount
    AND ABS(EXTRACT(EPOCH FROM (debit.posted_at - credit.posted_at))) <= 3 * 24 * 60 * 60
),
debit_best AS (
  -- For each debit, pick the closest credit; ties broken by smallest credit.id::text.
  -- Concern #4 fix: this ROW_NUMBER + filter rn=1 makes the pairing deterministic
  -- under ambiguity (one debit matching multiple credits same amount within window).
  SELECT debit_id, credit_id, delta_seconds
  FROM (
    SELECT
      debit_id, credit_id, delta_seconds,
      ROW_NUMBER() OVER (PARTITION BY debit_id ORDER BY delta_seconds ASC, credit_id::text ASC) AS rn
    FROM same_user_pairs
  ) ranked
  WHERE rn = 1
),
credit_best AS (
  -- Symmetric: for each credit, pick the closest debit; same tiebreaker.
  SELECT debit_id, credit_id, delta_seconds
  FROM (
    SELECT
      debit_id, credit_id, delta_seconds,
      ROW_NUMBER() OVER (PARTITION BY credit_id ORDER BY delta_seconds ASC, debit_id::text ASC) AS rn
    FROM same_user_pairs
  ) ranked
  WHERE rn = 1
),
mutual AS (
  -- Mutual best match: the pair (debit, credit) appears in BOTH best lists.
  -- This guarantees each transaction appears in at most one pair (1-to-1 invariant).
  -- Concern #4 closure: the existing single-side join produced multi-pair candidates;
  -- mutual best match collapses to a stable 1-to-1 mapping.
  SELECT db.debit_id, db.credit_id
  FROM debit_best db
  INNER JOIN credit_best cb USING (debit_id, credit_id)
),
updated AS (
  UPDATE ${transactions} t
  SET is_transfer = true,
      transfer_pair_id = CASE
        WHEN t.id = m.debit_id  THEN m.credit_id
        WHEN t.id = m.credit_id THEN m.debit_id
      END,
      updated_at = now()
  FROM mutual m
  WHERE t.id IN (m.debit_id, m.credit_id)
  RETURNING t.id
)
SELECT count(*)::int AS flagged FROM updated;
```
