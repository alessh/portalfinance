## Gap closure plan — Plan 02-07 overshoot (worker entry crashes on `server-only` CJS load)

**Source:** `.planning/phases/02-pluggy-ingestion/02-HUMAN-UAT.md` Test 1 BLOCKER (post 02-07/08/09).
**Premise:** Plan 02-07 added `import 'server-only';` to `src/lib/env.ts:25` and `src/lib/crypto.ts:18`. That package's runtime guard fires unconditionally under Node CJS — its `react-server` export condition only resolves under the Next.js webpack/turbopack alias OR with `node --conditions=react-server`. The worker entry (`src/jobs/worker.ts`) and several other tsx-direct scripts (`pnpm db:migrate`, `pnpm test:e2e`, ad-hoc `tsx -e ...`) load env.ts via Node CJS and crash at module load before any application code runs.

The repo already has TWO local workarounds — `tests/setup.ts` adds `vi.mock('server-only', () => ({}))` and `tests/fixtures/env-runner/env-runner.ts` stuffs `Module._cache[server-only]` with an empty exports object — but neither is reachable from `pnpm start:worker`. The fix must be in the source modules, not in test fixtures.

**Solution shape chosen — option (b): internal `assertServerOnly()` helper.**

Three options were on the table:

| Option | Approach | Why rejected / accepted |
|--------|---------|-------------------------|
| (a) | Detect `typeof window !== 'undefined'` inline at top of every server-only module | Rejected — fragile (every new server module must re-implement; copy-paste rot); no central place to harden against `navigator` polyfill / Node 22+ `navigator` global |
| **(b)** | **Single `src/lib/serverOnly.ts` helper exporting `assertServerOnly()`. Keep `import 'server-only';` at the TOP of that one file so Next.js's compile-time client-bundle guard still fires when a `'use client'` module transitively reaches it. Server modules call `assertServerOnly()` instead of importing the package directly.** | **Accepted — single source of truth; Node/tsx callers never hit the package's CJS throw because they walk `serverOnly.ts → assertServerOnly()` and the body uses `typeof window` checks; client bundles still fail the Next build because the leaf file `serverOnly.ts` keeps the package import** |
| (c) | Split `env.ts` into `envSchema.ts` (pure) + `envServer.ts` (server-only boundary), mirror what 02-07 did for `cpf.ts`/`cpfServer.ts` | Rejected for `env.ts` specifically — the schema-vs-loader split helps when client code legitimately needs the schema, but no client module needs `EnvSchema` (env validation is exclusively a boot-time server concern). The split would be ceremony with no client benefit, and would still need the `server-only` guard somewhere — pushing the original crash one module deeper |

**Compatibility constraint preserved.** `tests/unit/lib/cpf-client-isolation.test.ts` walks the static import graph rooted at `@/lib/cpf` and flags any path that hits `@/lib/env`, `@/lib/crypto`, or `@/lib/cpfServer`. After this plan:

- `cpfServer.ts` keeps its existing `import 'server-only';` (no change needed).
- The walker's two existing assertions both continue to pass: `cpf.ts` does not import `serverOnly`/`env`/`crypto`; `cpfServer.ts` does import `crypto` (sanity-check assertion).

**Companion fix — silence duplicate `.env not found`.** `pnpm start:worker` invokes tsx with `--env-file-if-exists=.env --env-file-if-exists=.env.local`. The new helper:

```typescript
// src/lib/serverOnly.ts (new)
/**
 * Server-only assertion helper. Replaces the previous direct
 * `import 'server-only';` in src/lib/env.ts and src/lib/crypto.ts (plan 02-07)
 * which crashed any non-RSC consumer (worker, db:migrate, ad-hoc tsx scripts).
 *
 * Two layers of defense:
 *   1. The literal `import 'server-only';` below — when a `'use client'` module
 *      transitively imports this file, Next.js's webpack/turbopack alias still
 *      fails the build with the canonical "needs server-only" error.
 *   2. The `assertServerOnly()` runtime check — when the same file is loaded
 *      from a true browser bundle that somehow slipped past the build guard
 *      (or in unit tests using `happy-dom`), the runtime throw fires.
 *
 * IMPORTANT: only this file imports the `server-only` package directly. All
 * other server-only modules call `assertServerOnly()` instead. That way:
 *   - Next.js client builds still fail at the leaf import.
 *   - Plain Node / tsx callers walk through this file, see no DOM, and pass.
 *
 * Plan 02-10 — gap closure for plan 02-07 overshoot (worker boot crash).
 */
import 'server-only';

const ERROR =
  'assertServerOnly: this module is server-only. It must NOT be imported ' +
  'from a Client Component, browser bundle, or any context where the global ' +
  '`window.document` exists. If you reached this from happy-dom in a unit ' +
  'test, mock @/lib/serverOnly to a no-op or move the assertion behind a ' +
  'lazy import.';

/**
 * Throws when called from a real browser-shaped context.
 *
 * The check is intentionally STRICT (`window` AND `window.document`) so that:
 *   - SSR-shim polyfills that do `globalThis.window = {}` (without a Document)
 *     do not falsely trip the guard.
 *   - Future Node versions that may add a global `window` object (analogous to
 *     `navigator` in Node 22+) without a real Document still pass.
 *   - Genuine browser bundles (where `window.document` is a `Document`
 *     instance) and happy-dom (which provides a `Document` instance) DO trip
 *     the guard — that is the desired behavior.
 */
export function assertServerOnly(): void {
  if (
    typeof window !== 'undefined' &&
    typeof (window as unknown as { document?: unknown }).document !== 'undefined'
  ) {
    throw new Error(ERROR);
  }
}
```

// tests/integration/observability/server-only-tsx-subprocess.test.ts
import { spawn } from 'node:child_process';
import { describe, it, expect } from 'vitest';
import { resolve } from 'node:path';

const REPO_ROOT = resolve(__dirname, '../../..');

const TSX_BIN = resolve(
  REPO_ROOT,
  'node_modules/.bin/tsx' + (process.platform === 'win32' ? '.cmd' : ''),
);

function runTsx(scriptPath: string, env: Record<string, string>): Promise<{ code: number; stdout: string; stderr: string; }> {
  return new Promise((resolveP) => {
    const child = spawn(TSX_BIN, [scriptPath], {
      cwd: REPO_ROOT,
      env: { ...process.env, ...env },
      stdio: ['ignore', 'pipe', 'pipe'],
      shell: process.platform === 'win32',  // Windows: tsx is a .cmd shim
    });
    let stdout = '';
    let stderr = '';
    child.stdout.on('data', (b) => { stdout += b.toString(); });
    child.stderr.on('data', (b) => { stderr += b.toString(); });
    child.on('close', (code) => resolveP({ code: code ?? -1, stdout, stderr }));
  });
}

const FIXTURE = resolve(REPO_ROOT, 'tests/fixtures/server-only/import-server-modules.ts');
// Plan 02-10 (INFO 1 of revision): use the absolute path to the tsx binary
// inside node_modules instead of `npx tsx`. Reasons:
//   - `npx` may not be on PATH in the spawn env on Windows (PowerShell vs.
//     cmd.exe vs. WSL).
//   - Resolving the binary directly is faster (no npx fetch / cache lookup).
//   - On Windows the binary is `tsx.cmd`; on POSIX it is `tsx` (no extension).
const TSX_BIN = resolve(
  REPO_ROOT,
  'node_modules/.bin/tsx' + (process.platform === 'win32' ? '.cmd' : ''),
);

interface SpawnResult {
  code: number;
  stdout: string;
  stderr: string;
}

function runTsx(env: Record<string, string>): Promise<SpawnResult> {
  return new Promise((resolveP) => {
    const child = spawn(TSX_BIN, [FIXTURE], {
      cwd: REPO_ROOT,
      env: { ...process.env, ...env },
      stdio: ['ignore', 'pipe', 'pipe'],
      // shell: true is needed on Windows so .cmd shims resolve correctly.
      shell: process.platform === 'win32',
    });
    let stdout = '';
    let stderr = '';
    child.stdout.on('data', (b) => {
      stdout += b.toString();
    });
    child.stderr.on('data', (b) => {
      stderr += b.toString();
    });
    child.on('close', (code) => {
      resolveP({ code: code ?? -1, stdout, stderr });
    });
  });
}

## Threat Model

**Trust boundaries (this plan):**

| Boundary | Description |
|----------|-------------|
| Server module → browser bundle | A `'use client'` module transitively importing `@/lib/serverOnly`, `@/lib/env`, or `@/lib/crypto` would expose server secrets / boot-time validation logic to the client; defended at compile time by the package import in `serverOnly.ts` + at run time by `assertServerOnly()` |
| Polyfill author → server runtime | A library that polyfills `globalThis.window` for SSR shimming could falsely trip `assertServerOnly()`, crashing the server on cold start |
| Future Node version → server runtime | Node 22+ added a global `navigator`; a future release might add a global `window` (analogous to the `Document` proposal in WHATWG) without a real Document |

**STRIDE register:**

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10-01 | I (Information disclosure) | `src/lib/env.ts` and `src/lib/crypto.ts` import surface | mitigate | `serverOnly.ts` keeps the literal `import 'server-only';` at the leaf — the Next.js bundler still fails any `'use client'` build that reaches it. Defense-in-depth: `cpfServer.ts` also keeps the package import. The cpf-client-isolation walker (updated in Task 4) flags any new client→server-only graph at unit-test time. |
| T-02-10-02 | E (Elevation of privilege via confused deputy) | `assertServerOnly()` runtime check | mitigate | The helper requires BOTH `typeof window !== 'undefined'` AND `typeof window.document !== 'undefined'` to throw. ## Rejected Alternatives

**Option (a) — Inline `typeof window` check at top of every server-only module.**
Rejected because every new server-only module would have to re-implement the same check and the same JSDoc rationale, leading to copy-paste rot. There would be no central place to harden against false positives (SSR-shim polyfills, future Node globals). The internal helper in option (b) consolidates the check and its threat model in one file — when the next Node version adds a new global, the fix lands in one place.

**Option (c) — Split `env.ts` into `envSchema.ts` (pure) + `envServer.ts` (server-only boundary).**
Rejected for `env.ts` specifically. The split-into-two-files pattern that plan 02-07 used for `cpf.ts → cpfServer.ts` made sense because client code legitimately needs `CPFSchema` for form validation. No client module needs `EnvSchema` — env validation is exclusively a boot-time server concern. Splitting would add a file with no client benefit and would still require the `server-only` guard somewhere, just one module deeper. It would also force every callsite that currently does `import { env } from '@/lib/env';` to change to `import { env } from '@/lib/envServer';`, which is mechanical churn with no payoff. The internal-helper approach in option (b) leaves the existing `@/lib/env` import surface unchanged and adds defense at a single new leaf module.
