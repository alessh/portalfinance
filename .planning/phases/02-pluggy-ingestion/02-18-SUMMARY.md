# Plan 02-18 — last_manual_sync_at cooldown anchor

Closes 02-REVIEWS.md Concern #12 (LOW).

## Outcome

Manual-sync cooldown is now anchored on a dedicated
`pluggy_items.last_manual_sync_at` column. The Phase 2 deliberate
semantics:

- Failed manual sync attempts do NOT cool down — the worker writes the
  column only inside the success path; a thrown error leaves it NULL.
- Recent automatic syncs (webhook, reconcile, first-connect, reconnect)
  do NOT block subsequent manual sync — they only update
  `last_synced_at`, leaving `last_manual_sync_at` untouched.

## Final route cooldown JSDoc (excerpt)

> Concern #12 deliberate semantics — the cooldown is anchored on
> `last_manual_sync_at`, NOT `last_synced_at`. This means:
> - A failed manual sync attempt does NOT cool down (the worker writes
>   `last_manual_sync_at` only after PluggyService calls succeed). The
>   user can retry; the connector-level Pluggy rate limit (PATCH /items
>   20/min) catches genuine abuse.
> - A recent automatic sync (webhook, reconcile, first-connect, reconnect)
>   does NOT block a subsequent manual sync. If the user just made a
>   purchase and wants to refresh now, a webhook 5 minutes ago is
>   irrelevant.
