# Phase 01.1: Infra Bootstrap (AWS sa-east-1 via Copilot) - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Stand up the AWS `sa-east-1` runtime topology via the AWS Copilot CLI, replacing the originally-planned Railway topology (Railway has no BR region, confirmed during Phase 1). Provisions the `web` (Next.js + API routes) and `worker` (pg-boss consumers) Fargate services, a private RDS Postgres 16 instance, secrets in SSM Parameter Store, an ALB fronted by Cloudflare, and a migrate job that runs `pnpm db:migrate` against RDS. Cuts the new AWS-native ops runbook to replace `docs/ops/railway-setup.md`. Flips Next.js to `output: 'standalone'` and pre-compiles the worker.

**No application-code changes beyond the build / entrypoint seam.** The Phase 1 schema, Auth.js wiring, pg-boss worker entry, SES client, and Sentry beforeSend scrubber all remain intact. LGPD-05 hard constraint (Brazilian territory) and every Phase 1 D-01..D-19 decision is honored.

**Requirements mapped:** OPS-01 (observability from day 1 — CloudWatch + Sentry EU), OPS-04 (runtime sandbox/prod assertion — preserved), LGPD-05 (data residency — `sa-east-1` only), SEC-02 (secrets at rest — KMS-encrypted SSM).

<decisions>
## Implementation Decisions

### Copilot Topology & Postgres

- **D-01:** **Postgres = RDS Postgres 16** provisioned as a Copilot storage addon. If the installed Copilot CLI version natively supports `copilot storage init --storage-type RDS-Postgres`, use it; otherwise write a CloudFormation addon template at `copilot/web/addons/rds-postgres.yml` (attaches outputs + IAM to the `web` service, and inbound rule is extended to the `worker` and `migrate` service security groups). **NOT Aurora Serverless v2** — minimum ACU cost (~USD 43/mo idle) is wasteful for pre-seed.
- **D-02:** **Single `prod` environment** provisioned in 01.1. Deferred: `staging` until first external beta user (Phase 5 territory). Rationale: Playwright E2E already runs locally against testcontainers; one ALB + one RDS is sufficient, and duplicate baseline saves USD 60–100/mo.
- **D-03:** **`web` = Copilot Load Balanced Web Service** (ALB + Fargate, public). Copilot manifest at `copilot/web/manifest.yml`. ACM public cert + TLS termination at ALB. Path-based routing enables future WAF rules.
- **D-04:** **`worker` = Copilot Backend Service** (no ALB, no public DNS). Fixed `count: 1`, no autoscaling — pg-boss per-user singleton-key pattern assumes a single replica. Horizontal scaling deferred to Phase 6. Copilot manifest at `copilot/worker/manifest.yml`.

### Secrets & DB Migration

- **D-05:** **Sensitive env vars live in SSM Parameter Store SecureString (KMS-encrypted)**, exposed via Copilot `secrets:` manifest blocks that resolve to `/copilot/portalfinance/prod/secrets/<NAME>`. Scope of "secret": `NEXTAUTH_SECRET`, `ENCRYPTION_KEY`, `CPF_HASH_PEPPER`, `SENTRY_DSN`, `SES_SMTP_PASSWORD` (or IAM-backed if using SDK v3 directly), `TURNSTILE_SECRET_KEY`. Non-secret config (`NODE_ENV`, `SENTRY_ENV`, `AWS_REGION`, `TURNSTILE_SITE_KEY` public) go under Copilot manifest `variables:`.
  - **Rotation story:** `aws ssm put-parameter --overwrite --name <path> --value <new> --type SecureString` followed by `copilot svc deploy` (forces task refresh).
  - `CPF_HASH_PEPPER` stays **distinct** from `ENCRYPTION_KEY` — both are separate SSM parameters (preserves Phase 1 RESEARCH.md Open Question #3 decision).
- **D-06:** **`DATABASE_URL` is composed at container start** by an entrypoint shim (`scripts/entrypoint.sh`) from RDS addon outputs (`DB_ENDPOINT`, `DB_PORT`, `DB_NAME`) + Secrets Manager credentials ARN (Copilot's RDS addon auto-manages the master password in Secrets Manager). Shim format: `export DATABASE_URL="postgresql://${DB_USER}:${DB_PASS}@${DB_ENDPOINT}:${DB_PORT}/${DB_NAME}?sslmode=require" && exec "$@"`. Shim is the Docker ENTRYPOINT; CMD stays per-service.
- **D-07:** **Schema migrations run via a dedicated Copilot Scheduled Job manifest** at `copilot/migrate/manifest.yml`. Reuses the same image as web/worker, `command: pnpm db:migrate`, same entrypoint shim so `DATABASE_URL` is composed identically. Triggered manually via `copilot job run --name migrate --env prod` immediately after `copilot svc deploy`. Idempotent (Drizzle tracks `drizzle.__migrations`). Runs in-VPC → no public RDS endpoint needed.
- **D-08:** **RDS is deployed in private subnets only**, no public endpoint. Developer-side ad-hoc access (`psql`, Drizzle Studio against prod) via `copilot svc exec --name web` into a running task. Bastion host deferred.

### Container & Next.js Build

- **D-09:** **Single multi-stage Dockerfile** at repo root (`Dockerfile`). Stages: `deps` (pnpm install), `builder` (pnpm build + pnpm build:worker), `runner` (copies `.next/standalone/`, `.next/static/`, `public/`, `dist/jobs/worker.js`, and `scripts/entrypoint.sh`). Copilot `web` manifest sets `command: node .next/standalone/server.js`; `worker` manifest sets `command: node dist/jobs/worker.js`; `migrate` manifest sets `command: pnpm db:migrate` (keeps tsx only in the migrate image if needed — see D-11). **Same image tag across all three services** (one build, three deployments).
- **D-10:** **Flip `next.config.ts` to `output: 'standalone'`** in this phase. Aligns `package.json` `start:web` script to `node .next/standalone/server.js` so local start matches the container. Closes the Phase 1 deferred item from plan 01-00. Reduces image size (~500 MB → ~150 MB) and removes the Next 16 standalone warning on boot.
- **D-11:** **Pre-compile the worker** to `dist/jobs/worker.js` via `tsup` (or esbuild) as a new `build:worker` npm script. Removes tsx from the prod runner image. Dev-time `pnpm start:worker` continues to use tsx. For the migrate job: `pnpm db:migrate` imports `src/db/migrate.ts`, which uses tsx today — either (a) include tsx as a prod dep so the migrate image runs it, or (b) pre-compile migrate.ts to `dist/db/migrate.js` and add a `db:migrate:prod` script. Planner picks the lowest-friction option.
- **D-12:** **Base image = `public.ecr.aws/docker/library/node:22-alpine`** (AWS ECR Public proxy — avoids Docker Hub rate limits in CI and on the dev machine). Builder stage installs `python3 make g++` for `argon2` native compile, then discards; runner stage ships only the precompiled binary. Final image size target: < 200 MB.

### Edge / Cloudflare

- **D-13:** **Cloudflare orange-cloud DNS in front of the public ALB.** Cloudflare manages `portalfinance.app` DNS authoritatively (already the case in Phase 1 for SES DKIM/SPF). Proxied CNAME records `@` and `www` and `api` (if split) point to the ALB DNS name. TLS mode = **Full (Strict)** — Cloudflare terminates at the edge and re-encrypts to the ALB using the ACM public cert. Gains: CDN, DDoS, WAF, Turnstile, rate-limiting at the edge.
- **D-14:** **ACM public cert on the ALB** for `portalfinance.app` + `*.portalfinance.app`, DNS-validated via Cloudflare DNS (manual one-time CNAME creation). Auto-renewed by AWS. No Origin CA cert (preserves ability for Pluggy sandbox webhook replay or direct health checks if ever needed, though production traffic always flows through Cloudflare).
- **D-15:** **All routes — including webhooks (`/api/webhook/pluggy`, `/api/webhook/asaas`, `/api/webhook/ses-bounce`) — flow through Cloudflare → ALB.** Cloudflare rate-limit rules protect webhook endpoints; IP allowlists for known provider ranges (Pluggy, ASAAS, AWS SNS) optional. Single ingress path keeps logs and observability uniform.
- **D-16:** **Cloudflare Turnstile** (Phase 1 D-07) verified in 01.1 — confirm the site key + secret key both resolve through the Cloudflare-proxied domain, update the Turnstile site configuration if the hostname changes. **R2 object storage deferred** (no consumer in Phase 2/3/4; revisit in Phase 5 for NFS-e PDFs).

### AWS Account & IAM Access

- **D-17:** **Single AWS account** — the same account that already holds SES `sa-east-1` (Phase 1 D-09). **IAM Identity Center SSO** (AWS SSO) with an `AdministratorAccess` permission set assigned to the dev user. Copilot CLI authenticates via `aws sso login --profile portalfinance-prod`. **No long-lived IAM access keys** — reduces breach surface. Organizations / separate `prod` account deferred to Phase 6.

### VPC & Private Connectivity

- **D-18:** **VPC endpoints + one NAT gateway** for egress. Interface endpoints (≈ USD 7/mo each) for: `ssm`, `ssmmessages` (ECS Exec), `secretsmanager`, `ecr.api`, `ecr.dkr`, `sts`, `logs` (CloudWatch). Gateway endpoint (free) for `s3` (ECR image layer storage). NAT gateway kept (≈ USD 33/mo + data egress) for external HTTPS traffic to Pluggy, ASAAS, Gemini, Sentry EU, SES. Baseline connectivity cost ~USD 40/mo.
  - **Why both:** AWS-service traffic stays on the private AWS backbone (faster + no NAT egress charges). External traffic still needs NAT. Deleting the NAT to save USD 33 is not worth the multi-service rewire.

### Fargate Sizing & RDS Class

- **D-19:** **web Fargate task = 0.5 vCPU / 1 GB RAM; worker = 0.25 vCPU / 512 MB; RDS = db.t4g.micro** (2 vCPU / 1 GB, ARM Graviton2, Multi-AZ **disabled** in v1). Storage = 20 GB gp3 general purpose. Target baseline ~USD 90–120/mo all-in (Fargate + RDS + ALB + NAT + VPC endpoints + CloudWatch + Cloudflare Free). Upgrade RDS to `db.t4g.small` when real user volume stresses IOPS (~Phase 5 billing + dashboard traffic). Multi-AZ RDS deferred to Phase 6 hardening.

### CI/CD Deploy Flow

- **D-20:** **Manual `copilot deploy` from developer laptop** is the deploy path in 01.1. Developer runs `copilot svc deploy --name web --env prod` + `copilot svc deploy --name worker --env prod` + `copilot job run --name migrate --env prod`. No GitHub Actions pipeline, no CodePipeline. **Automated CI/CD deferred** to a dedicated ops phase (post-Phase 6) once deploy frequency justifies it. This phase optimizes for bootstrap speed, not deploy ergonomics.

### ECR Image Lifecycle

- **D-21:** **Git SHA tags + ECR lifecycle policy keeping last 20 images** per Copilot-auto-created repo (one per service). Rollback = `copilot svc deploy --name web --tag <old-sha>`. Environment pointer tags (`web:prod`) are fine but the SHA is the source of truth.

### Cost Guardrails & Tagging

- **D-22:** **Standard Copilot tags + custom project tags** on every resource: Copilot auto-adds `copilot-application`, `copilot-environment`, `copilot-service`; Copilot manifest `tags:` block adds `Project=PortalFinance`, `Phase=01.1`. **AWS Budgets alert at 80% of USD 150/mo** (email-only action) on the Portal Finance linked account. Cost Explorer + tag-based cost allocation enabled. No auto-shutdown actions.

### Application Logs

- **D-23:** **Container stdout → CloudWatch Logs (Copilot default)** with 30-day retention per log group. Structured JSON from Pino (Phase 1 wiring) flows unchanged. **Sentry EU** (`de.sentry.io`) continues to receive errors only via the existing `beforeSend` scrubber. **No log shipping / aggregator** in 01.1 — CloudWatch Insights + Sentry covers v1 volume. Revisit in Phase 6 when oncall dashboards mature.

### Health Checks

- **D-24:** **New `/api/health` Next.js route** at `src/app/api/health/route.ts` returning `{ status: "ok" }` + 200. Used by ALB target group health check (path `/api/health`, interval 30 s, healthy threshold 2, unhealthy 3) and ECS container health check. **Do not** point the health check at `/` — that would render the demo dashboard (D-03 of Phase 1) on every probe and pollute access logs.

### Claude's Discretion

- **Copilot app name**: `portalfinance` (single app, three services + one job).
- **VPC CIDR**: `10.20.0.0/16` with 2 public + 2 private subnets across two `sa-east-1` AZs (mandatory for future RDS Multi-AZ without VPC rewire). Copilot's `env init` default is acceptable if the chosen CIDR doesn't collide with developer VPNs.
- **Copilot CLI version pin**: latest stable at bootstrap time; document the pin in the new runbook.
- **Entrypoint shim language**: `scripts/entrypoint.sh` (POSIX `sh`, not bash, so alpine's `/bin/sh` runs it directly).
- **tsup vs esbuild for worker compile**: planner's call — tsup's one-liner config is friendlier; esbuild is more powerful. Either is fine.
- **ECS Exec enablement**: enabled on all three services from day 1 (needed for in-VPC psql and on-call access).
- **CloudWatch log group names**: Copilot defaults (`/copilot/portalfinance-prod-web`, etc.) are fine.
- **Dockerfile `.dockerignore`**: must exclude `.next/`, `node_modules/`, `.planning/`, `playwright-report/`, `test-results/`, `.env.local`.
- **RDS backup window**: Copilot default (automated backups, 7-day retention) for 01.1; Multi-AZ and cross-region deferred to Phase 6.
- **Security group layering**: web SG → ALB SG; worker SG and migrate SG → RDS SG (inbound 5432). Copilot default addon output handles most of this; addon YAML only adds the worker + migrate ingress rules.
- **Required runtime env vars NOT stored in SSM**: `NODE_ENV`, `AWS_REGION` (= `sa-east-1`), `SENTRY_ENV`, `PORT` (Copilot sets to 80 by default — align Next standalone server to read `PORT`), `TURNSTILE_SITE_KEY` (public), `AWS_DEFAULT_REGION`.
- **Existing `.env.local` is NOT shipped** — it's in `.dockerignore`; prod values come only from SSM + manifest `variables:`.

<specifics>
## Specific Ideas

- **Entrypoint shim invocation order**: shim must run `aws ssm get-parameters --with-decryption` (or rely on Copilot `secrets:` injection) AND compose `DATABASE_URL` BEFORE `exec`-ing the final `node` process. If the Copilot task role already exposes secrets as env vars (which it does when `secrets:` is set), the shim only needs to compose `DATABASE_URL` from the RDS outputs + Secrets Manager password.
- **Preserve Phase 1 SES domain identity**: `no-reply@portalfinance.app` (D-11 of Phase 1) is already a verified SES identity in `sa-east-1`. The AWS account holding SES identity is the same account provisioning Copilot resources — DNS records (DKIM, SPF, DMARC `p=none` with rua) on Cloudflare remain as-is.
- **Migrate job retry behavior**: migrate job is idempotent (Drizzle `__migrations` table) — safe to invoke multiple times. If a deploy fails mid-migration, the next run picks up where the last left off; document this in the new runbook.
- **Same image for all three services** is an optimization: build once, tag once, deploy three places. Copilot supports this via `copilot svc deploy --name web --tag <sha>` + `worker` + `migrate` all pointing at the same ECR image.
- **Rollback story**: keep the previous Copilot deploy revision in ECS service history. `copilot svc deploy --tag <prior-sha>` reverts. RDS schema rollbacks require writing a compensating Drizzle migration, never manual SQL on prod.

</specifics>
