## Copilot Topology & Postgres

### Postgres hosting

| Option | Description | Selected |
|--------|-------------|----------|
| RDS Postgres 16 (manual CF addon) | Dedicated RDS instance, Postgres 16, Copilot CloudFormation addon at `copilot/<svc>/addons/db.yml`, predictable cost, full instance-class control. | |
| Aurora Serverless v2 Postgres | Copilot's blessed path via `copilot storage init --storage-type Aurora`. Min 0.5 ACU idle ≈ USD 43/mo. Smooth scaling but overkill at pre-seed. | |
| RDS Postgres via Copilot storage (if supported) | Verify Copilot CLI supports `--storage-type RDS-Postgres` natively; otherwise fall back to option 1. | ✓ |

**User's choice:** RDS Postgres via Copilot storage (if supported) — fallback to manual CloudFormation addon if the installed Copilot CLI version doesn't ship that storage type.

### Environments from day 1

| Option | Description | Selected |
|--------|-------------|----------|
| `prod` only | Single environment; staging deferred. Saves USD 60–100/mo. | ✓ |
| `prod` + `staging` | Two environments with duplicated ALB + RDS. | |
| `dev` + `staging` + `prod` | Full three-tier; overkill at v1. | |

**User's choice:** `prod` only.

### Web compute pattern

| Option | Description | Selected |
|--------|-------------|----------|
| Load Balanced Web Service (ALB + Fargate) | Copilot standard pattern; ALB + ACM + Fargate task. | ✓ |
| Request-Driven Web Service (App Runner) | Simpler but less control; `sa-east-1` availability historically spotty. | |

**User's choice:** Load Balanced Web Service (ALB + Fargate).

### Worker scaling

| Option | Description | Selected |
|--------|-------------|----------|
| Backend Service, fixed count = 1 | Matches pg-boss per-user singleton-key assumption. | ✓ |
| Backend Service, autoscaling 1–2 | Adds complexity without v1 benefit. | |

**User's choice:** Backend Service, fixed count = 1.

## Secrets + DB Migration

### Sensitive env-var storage

| Option | Description | Selected |
|--------|-------------|----------|
| Copilot `secrets:` → SSM Parameter Store (SecureString) | KMS-encrypted, free tier, IAM-scoped to task role, `aws ssm put-parameter --overwrite` rotation. | ✓ |
| AWS Secrets Manager | Built-in rotation + versioning; ~USD 0.40/mo per secret. Overkill for symmetric app keys. | |
| Hybrid: Copilot secrets for app keys, RDS creds auto-managed | Copilot's RDS addon auto-manages the password in Secrets Manager — user accepted this as part of the chosen option (see CONTEXT D-05 / D-06). | |

**User's choice:** Copilot `secrets:` → SSM Parameter Store (SecureString). (RDS-password Secrets Manager integration is included implicitly via Copilot's RDS addon.)

### `DATABASE_URL` composition

| Option | Description | Selected |
|--------|-------------|----------|
| Composed at container start from RDS addon outputs | `scripts/entrypoint.sh` resolves RDS outputs + Secrets Manager password → `DATABASE_URL` → `exec` Node. | ✓ |
| Static `DATABASE_URL` stored as its own SSM SecureString | Simpler runtime; manual update on every RDS rotation. | |
| Lazy construction in `src/db/index.ts` | Keeps image generic; requires a code change. | |

**User's choice:** Composed at container start from RDS addon outputs.

### Migration runner

| Option | Description | Selected |
|--------|-------------|----------|
| Copilot job, run before each deploy | `copilot/migrate/manifest.yml` Scheduled Job, same image, CMD `pnpm db:migrate`, in-VPC, idempotent. | ✓ |
| ECS RunTask from GitHub Actions / local step | Post-deploy shell step; drifts from Copilot knowledge. | |
| Manual from developer box via public RDS endpoint | Simplest bootstrap but violates VPC isolation. | |

**User's choice:** Copilot job, run before each deploy.

### RDS accessibility

| Option | Description | Selected |
|--------|-------------|----------|
| Private subnet only (VPC-internal) | Strongest posture; dev access via `copilot svc exec`. | ✓ |
| Private + SSH bastion | Adds USD 3/mo + patch burden. | |
| Public RDS endpoint (IP-whitelist) | Weakest security. | |

**User's choice:** Private subnet only (VPC-internal).

## Dockerfile & Next.js Build

### Dockerfile strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Single Dockerfile, two manifests override CMD | One multi-stage build, three services share the image. | ✓ |
| Two Dockerfiles (web + worker) | Doubles build time, duplicates deps. | |
| Single container runs both processes | Mixes failure domains. | |

**User's choice:** Single Dockerfile, two service manifests override CMD.

### Next.js build mode

| Option | Description | Selected |
|--------|-------------|----------|
| Flip to `output: 'standalone'` + `node server.js` | Closes STATE.md deferred item; ~3–5× smaller image. | ✓ |
| Keep `next start` (non-standalone) | Larger image, boot warning. | |

**User's choice:** Flip to `output: 'standalone'` + `node server.js`.

### Worker entrypoint

| Option | Description | Selected |
|--------|-------------|----------|
| Ship tsx runtime, CMD `tsx src/jobs/worker.ts` | Matches dev verbatim; tsx + TS source in prod (~15 MB overhead). | |
| Pre-compile worker, CMD `node dist/jobs/worker.js` | Faster startup, smaller image; requires tsup/esbuild script. | ✓ |

**User's choice:** Pre-compile worker with tsup/swc → `node dist/jobs/worker.js`.

### Base image

| Option | Description | Selected |
|--------|-------------|----------|
| `node:22-alpine` (Docker Hub) | Tiny, LTS; musl + argon2 native build quirk in builder stage. | |
| `node:22-slim` (Debian) | Larger (~70 MB more) but fewer native-module surprises. | |
| `public.ecr.aws/.../node:22-alpine` | Same as option 1 from AWS ECR Public — no Docker Hub rate limits. | ✓ |

**User's choice:** AWS-published `public.ecr.aws/docker/library/node:22-alpine`.

## Edge: Cloudflare Front

### Cloudflare topology

| Option | Description | Selected |
|--------|-------------|----------|
| Cloudflare orange-cloud DNS in front of public ALB | CF terminates TLS, re-encrypts to ALB (Full Strict). Gains WAF, DDoS, CDN, Turnstile. | ✓ |
| Cloudflare Tunnel → private ALB | Stronger posture; single ingress path (CF-only) is an availability risk. | |
| Skip Cloudflare, expose ALB directly | Loses WAF/Turnstile — strands Phase 1 D-07. | |

**User's choice:** Cloudflare orange-cloud DNS in front of public ALB.

### ALB TLS

| Option | Description | Selected |
|--------|-------------|----------|
| ACM public cert (`portalfinance.app` + wildcard) | DNS-validated via Cloudflare; auto-renews. | ✓ |
| Cloudflare Origin CA cert | Tighter coupling to CF; non-CF clients can't hit origin. | |

**User's choice:** AWS ACM public cert.

### Webhook path

| Option | Description | Selected |
|--------|-------------|----------|
| Through Cloudflare → ALB (same as app traffic) | Consistent ingress; CF protects webhooks. | ✓ |
| Direct ALB endpoint for webhooks, CF for app | Loses CF anti-abuse on the most-attacked endpoints. | |

**User's choice:** Through Cloudflare → ALB.

### Turnstile / R2 scope

| Option | Description | Selected |
|--------|-------------|----------|
| Turnstile: enable now; R2: defer | Turnstile is Phase 1 D-07; R2 has no v1 consumer. | ✓ |
| Both now | Pre-work with no consumer. | |
| Defer both | Strands Phase 1 decision. | |

**User's choice:** Turnstile now; R2 deferred.

## Fargate Sizing + RDS Class

| Option | Description | Selected |
|--------|-------------|----------|
| web 0.5 / 1 GB, worker 0.25 / 512 MB, RDS db.t4g.micro | Baseline ~USD 90–120/mo all-in. | ✓ |
