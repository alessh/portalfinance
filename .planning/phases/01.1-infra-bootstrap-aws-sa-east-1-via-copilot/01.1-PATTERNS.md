### `src/app/api/health/route.ts` (route, request-response)

**Analog:** `src/app/api/auth/logout/route.ts`

**Key observations from analog:**
- Explicit `export const runtime = 'nodejs'` at top (every existing route in `src/app/api/**` declares it — grep confirms 11/11).
- Named function `export async function POST/GET` with typed `NextRequest` parameter.
- Body doc-comment block explaining the plan ID, requirement trace, and purpose.
- Returns `NextResponse.json(...)` or `new Response(...)` with explicit 200 status.

**Header + runtime pattern** (`src/app/api/auth/logout/route.ts` lines 1-10):
```typescript
/**
 * POST /api/auth/logout — AUTH-03 server-side session invalidation.
 *
 * [purpose + architecture summary]
 */

import { NextResponse, type NextRequest } from 'next/server';
```

**Tiny-response pattern** (`src/app/api/auth/logout/route.ts` lines 35-45):
```typescript
const res = NextResponse.json({ ok: true }, { status: 200 });
// ... cookie manipulation omitted ...
return res;
```

**Recommended shape for the new health route:**
```typescript
/**
 * GET /api/health — D-24 / OPS-01 ALB + ECS health probe.
 *
 * Target group health check (interval 30s, healthy 2, unhealthy 3).
 * Must NOT touch the DB — Phase 1 lazy Drizzle client would panic if
 * DATABASE_URL were missing, and probe latency must stay < 5s (Copilot
 * manifest timeout). This endpoint intentionally returns static JSON.
 */
export const runtime = 'nodejs';

import { NextResponse } from 'next/server';

export async function GET(): Promise<Response> {
  return NextResponse.json({ status: 'ok' }, { status: 200 });
}
```

**Do NOT copy from analog:**
- DB access (logout DELETEs a session row — health route must stay DB-free).
- Cookie logic.
- Audit log side-effect (`recordAudit`).

### `scripts/entrypoint.sh` (runtime-shim)

**No shell-script analog in repo.** The only `scripts/` file is `scripts/run-e2e.ts` (TypeScript, not POSIX sh).

**Canonical reference:** **RESEARCH.md § Pattern 4: Entrypoint Shim (POSIX sh)** (lines 598-631 of `01.1-RESEARCH.md`).

**Cross-cutting rules (from project conventions + CONTEXT.md D-06):**
- POSIX `sh` only (shebang `#!/bin/sh`, NOT bash) — alpine's `/bin/sh` is busybox.
- `set -eu` at top.
- `exec "$@"` as last line (not `$@` directly) so the CMD becomes PID 1 (or under tini).
- Uses AWS CLI v2 (baked into runner stage via `apk add --no-cache aws-cli`).
- Comments in US-English.

### `Dockerfile` (build)

**No analog** — repo has never had a Dockerfile.

**Canonical reference:** **RESEARCH.md § Pattern 5: Multi-Stage Dockerfile** (lines 633-693).

**Cross-cutting rules:**
- Base: `public.ecr.aws/docker/library/node:22-alpine` (D-12 — AWS ECR Public proxy, not Docker Hub).
- Three stages: `deps`, `builder`, `runner`.
- Runner installs `aws-cli` (needed by `scripts/entrypoint.sh`), `ca-certificates`, `tini`.
- `ENTRYPOINT ["/sbin/tini", "--", "/entrypoint.sh"]` + per-service `CMD`.

- Non-root user `nextjs:nodejs` (1001:1001).
- `argon2` recompile in builder stage only (`apk add python3 make g++`), runner ships the `.node` binary.

## Shared Patterns

### Pattern: File-top doc comment explaining plan ID + requirement trace

**Source:** every `src/**/*.ts` file in Phase 1 (example: `src/app/api/webhooks/ses/bounces/route.ts` lines 1-19, `src/app/api/auth/logout/route.ts` lines 1-7, `drizzle.config.ts` lines 1-15).

**Apply to:** EVERY new `.ts` / `.tsx` / `.sh` / YAML file in 01.1.

**Canonical shape:**
```
/**
 * <one-line summary of what this file does>
 *
 * <Plan reference — e.g., "Plan 01.1-XX"> / <Requirement — e.g., "OPS-01">.
 *
 * <Architecture / flow summary (2-4 lines)>.
 *
 * CRITICAL invariants: (optional but encouraged for infra files)
 *   1. ...
 *   2. ...
 */
```

Example from `src/app/api/webhooks/ses/bounces/route.ts` lines 1-19:
```typescript
/**
 * SES bounce / complaint SNS webhook receiver.
 *
 * Plan 01-04 — T-WH-REPLAY + T-WH-FORGE mitigations (D-15).
 *
 * Architecture (CONTEXT.md):
 *   SNS topic → HTTPS POST to this route → idempotent webhook_events INSERT
 *   → enqueue ses.bounce pg-boss job → return 200 in < 200 ms
 *
 * CRITICAL invariants:
 *   1. verifySnsMessage MUST precede any DB insert (T-WH-FORGE).
 *   2. webhook_events uses UNIQUE(source, event_id) + onConflictDoNothing — ...
 *   3. Return 200 < 200 ms after insert + enqueue (SNS retries on non-2xx).
 */
```

### Pattern: Snake_case for local variables that hold "data" (project naming convention)

**Source:** `src/app/api/webhooks/ses/bounces/route.ts` lines 40, 71, 83 (`sig_ok`, `was_duplicate`, `latency_ms`). Also `CLAUDE.md` global convention table.

**Apply to:** every new `.ts` / `.sh` file (shell-script env-var naming too — `DB_USER`, `DB_PASS`, `DB_SECRET_ARN`).

**Rules summary:**
- Functions: `camelCase` (`getAvailableJobs`).
- Local variables / private members: `snake_case` (`sig_ok`, `webhook_event_id`).
- Loop-scoped or throwaway locals: `camelCase` acceptable per global convention.
- Constants: `UPPER_SNAKE` (`SESSION_COOKIE_NAME` in `logout/route.ts` line 16-19, `MAX_RETRIES`).
- Acronyms always uppercase: `AWS`, `RDS`, `SSM`, `ALB`, `VPC`, `JSON`, `API`, `HTTP`, `HTTPS`, `SQL`, `CSV`, `XML`, `CPF`, `LGPD`, `DNS`, `TLS`.

### Pattern: US-English everywhere (comments, variables, commit messages, docs)

**Source:** global `~/.claude/CLAUDE.md` + project `./CLAUDE.md`. Every existing file in the repo observes this.

**Apply to:** every new file in 01.1 — including Portuguese-adjacent strings like SES justification (which is already EN_US in `ses-production-access.md`).

### Pattern: Structured JSON logging via Pino (NOT console.log)

**Source:** `src/app/api/webhooks/ses/bounces/route.ts` lines 42-46, 54, 78-83 — every log line goes through `import { logger } from '@/lib/logger'` with structured fields `{ event, latency_ms, ... }`.

**Apply to:** any new route or script that logs. The health route should NOT log per-request (noise) but any error path must use `logger`.

**Excerpt:**
```typescript
import { logger } from '@/lib/logger';

logger.info({
  event: 'ses_bounce_received',
  messageId: body.MessageId,
  latency_ms: Date.now() - start,
  was_duplicate,
}, 'SES bounce webhook processed');
```

### Pattern: Lazy DB client — never assume `DATABASE_URL` at build time

**Source:** `src/db/index.ts` (Phase 1 Plan 01-02 decision, referenced in CONTEXT.md § Existing Code Insights line 159-160).

**Apply to:** Dockerfile builder stage. The builder runs `pnpm build` WITHOUT a real `DATABASE_URL` — Next 16 page-data collection accepts the placeholder because `src/db/index.ts` defers the actual connection to first query.

**Implication for `scripts/entrypoint.sh`:** the shim MUST run and `export DATABASE_URL=...` BEFORE `exec`-ing `node`, because `src/lib/env.ts` re-validates at module load and will throw if `DATABASE_URL` is missing at that moment (not at build time).

### Pattern: Ignore-file structure (mirror `.gitignore` sections)

**Source:** `.gitignore` lines 1-44.

**Apply to:** `.dockerignore` — reuse section headings (`# Dependencies`, `# Next.js`, `# Build output`, `# Environment`, `# Test artifacts`, `# Tooling caches`, `# Logs`, `# Editor / OS`) for consistency; add `# Repo-only — never ship to image` section for `.planning/`, `copilot/`, `docs/`, `.git/`.
