<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer SSO session -> AWS CloudFormation | Admin-level actions; scoped by STS session TTL |
| local `copilot` CLI -> AWS APIs | Uses SSO-cached credentials; no long-lived keys touched |
| SSM SecureString write -> KMS | `aws/ssm` managed key by default; KMS access is implicit via IAM |
| deployed tasks -> RDS | Security group ingress only from the Copilot environment SG (which web/worker/migrate are members of) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.1-RDS-PUB | Information Disclosure | RDS endpoint reachability | mitigate | PubliclyAccessible: false declared in CFN (Plan 01.1-04); Task 1 acceptance greps the live describe-db-instances response for false |
| T-01.1-SSM | Information Disclosure + Tampering | SSM parameter types | mitigate | `copilot secret init` always writes SecureString; Task 2 acceptance greps describe-parameters for Type=SecureString on every parameter |
| T-01.1-IAM | Spoofing | developer credentials during deploy | mitigate | Task 3 uses `AWS_PROFILE=portalfinance-prod` (SSO) throughout -- no IAM user, no access keys |
| T-DEPLOY-ROLLBACK | Denial of Service | mid-deploy failure | mitigate | Copilot retains previous task-definition revision; if svc deploy fails the ALB stays on the prior healthy task set. Explicit rollback via `copilot svc deploy --tag <prior-sha>` documented in Plan 01.1-08 runbook |
| T-LOG-RETENTION | Information Disclosure | CloudWatch logs beyond 30 days | mitigate | Manifest logging.retention: 30 on all three services; Task 3 acceptance greps live log-groups describe for retentionInDays=30 only |
</threat_model>
